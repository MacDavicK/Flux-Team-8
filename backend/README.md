# Flux Backend

AI-powered accountability coach backend — FastAPI + LangGraph + LiteLLM.

---

## Prerequisites

- [Docker](https://docs.docker.com/get-docker/) and Docker Compose
- [uv](https://github.com/astral-sh/uv) (for local development without Docker)
- Python 3.12+
- A [Supabase](https://supabase.com) project
- An [OpenRouter](https://openrouter.ai) API key
- A [Pinecone](https://app.pinecone.io) account (serverless index — free tier works)
- A [Twilio](https://twilio.com) account (WhatsApp + Voice + Verify)
- (Optional) A [Google Cloud](https://console.cloud.google.com) OAuth 2.0 Client ID for Google login

## Quickstart

### 1. Clone the repo

```bash
git clone <repo-url>
cd backend
```

### 2. Start the dev environment

Run `setup.sh` from the **repo root** — it checks dependencies, configures `.env` interactively (skipping values already set), applies all database migrations via Docker, starts the full backend stack, and launches the frontend dev server:

```bash
bash ../setup.sh
```

This starts:
- **api** — FastAPI app on `http://localhost:8000`
- **notifier** — background notification worker
- **redis** — rate limiter and job store
- **ngrok** — tunnel for Twilio webhooks (local dev)

Swagger UI: `http://localhost:8000/docs`
ngrok inspector: `http://localhost:4040`

> **Production:** Do not set `NGROK_AUTHTOKEN` or `NGROK_DOMAIN`. Set `TWILIO_WEBHOOK_BASE_URL` to your deployed API URL instead.

### 3. Tear down the dev environment

Press **Ctrl-C** to stop. `setup.sh` automatically stops all containers on exit.

To tear down manually:

```bash
# Stop containers only
docker compose --profile ngrok down --remove-orphans

# Full teardown (also remove volumes and build cache)
docker compose --profile ngrok down --volumes --remove-orphans
docker builder prune --force
```

## Google OAuth Setup (One-Time)

To enable "Continue with Google" on the login screen:

### 1. Create a Google OAuth Client

1. Go to [Google Cloud Console → Credentials](https://console.cloud.google.com/apis/credentials)
2. Create → **OAuth 2.0 Client ID** → Application type: **Web application**
3. Add Authorized redirect URIs:
   - `https://<your-supabase-project>.supabase.co/auth/v1/callback` (Supabase-side exchange)
   - `http://localhost:3000/auth/callback` (local dev — TanStack Start callback route)
4. Copy the **Client ID** and **Client Secret**

### 2. Configure Supabase

1. **Supabase Dashboard → Authentication → Providers → Google**
   - Enable the Google provider
   - Paste in the Client ID and Client Secret from Step 1

2. **Supabase Dashboard → Authentication → URL Configuration**
   - Set **Site URL**: `http://localhost:3000` (or your production URL)
   - Add to **Additional Redirect URLs**: `http://localhost:3000/auth/callback`

### 3. Frontend Environment

Add to `frontend/.env`:
```
VITE_SUPABASE_URL=https://<your-project>.supabase.co
VITE_SUPABASE_ANON_KEY=<your-anon-key>
```

> **Note:** The backend does not require changes — it already validates Supabase JWTs regardless of whether the user authenticated via email/password or Google OAuth.

## Architecture Overview

- **API Layer**: FastAPI with JWT auth (Supabase), per-user rate limiting (slowapi + Redis), structured logging (structlog)
- **Agent Layer**: LangGraph graph with Orchestrator → Goal Planner → Classifier + Scheduler + Pattern Observer + RAG Retriever (parallel fan-out) → Save Tasks
- **RAG Pipeline**: Health/fitness goals trigger `rag_retriever_node` in the fan-out. It embeds the goal query, fetches the top-k matching chunks from Pinecone, and injects expert context into the Goal Planner prompt. Corpus: 30 curated articles in `backend/articles/` (weight loss, nutrition, strength, cardio, behavioral).
- **LLM**: All models accessed via OpenRouter (LiteLLM), with 3-tier fallbacks and monthly token budgets
- **Notifications**: Push (Web Push/VAPID) → WhatsApp (Twilio) → Voice call (Twilio DTMF) escalation chain, with atomic CAS to prevent duplicate dispatches
- **Database**: Supabase (PostgreSQL) with RLS enabled on all user-data tables; asyncpg for direct connections

## RAG Operations

### First-time corpus ingestion

`setup.sh` handles ingestion automatically (Step 5). If you need to trigger it manually:

```bash
# Ingest articles into Pinecone (APP_ENV=development, no auth required)
curl -X POST http://localhost:8000/api/v1/rag/ingest
```

The endpoint is idempotent: calling it again clears and re-ingests all articles.

### Debug search

Test retrieval quality without going through the full agent pipeline:

```bash
# Requires a valid JWT (or APP_ENV=development bypasses auth via the standard dev bypass)
curl "http://localhost:8000/api/v1/rag/search?q=how+to+lose+weight+gradually" \
  -H "Authorization: Bearer <token>"
```

Returns the raw top-k chunks with similarity scores.

### Adding articles to the corpus

1. Add a `.txt` file to `backend/articles/` following the header format:
   ```
   Title: <title> | Source: <url>
   Category: <category> | Authority: <authority>
   ---
   <body text>
   ```
2. Re-run ingestion: `curl -X POST http://localhost:8000/api/v1/rag/ingest`

### Adding a new RAG category (e.g., mental health articles)

1. Add articles in `backend/articles/` with the new category label.
2. In `backend/app/config.py`, add the relevant classifier tag to `rag_trigger_tags`:
   ```python
   rag_trigger_tags: list[str] = ["Health", "Fitness", "Nutrition", "Mental Health", "Mindfulness"]
   ```
3. Re-run ingestion to pick up the new articles.
